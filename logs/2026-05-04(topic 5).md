## 🔥 Why This Matters (Real World)

Dicts are everywhere in backend systems — API request/response bodies are dicts, database query results are dicts, config objects are dicts, session data is a dict. Understanding how they work internally is critical because a wrong key lookup pattern, unhandled missing keys, or poor dict construction at scale can silently destroy your API's performance.

## ❓ What I THOUGHT I Knew (Before Today)

I thought dict just stores key-value pairs and accesses values using the key. I had no idea HOW it finds the value — I assumed it scans through all keys to find a match, like a list. I also didn't know why list can't be a dict key or what defaultdict/Counter actually solve internally.

---

## 💡 What I Actually Learned (In MY Words)

When I do `d["name"] = "alice"`, Python doesn't just "store" it somewhere random. It runs `hash("name")` to get a big number, then uses `% 8` (modulo) to shrink that number into a valid slot index (0–7). It stores `"alice"` at that exact slot in an internal array.

When I lookup `d["name"]`, Python does the same math — same key → same hash → same slot → direct access. No searching. That's why it's O(1) always, even with a million keys.

The key insight: **dict doesn't search, it calculates.**

---

## 🔍 The DEEP Answer (Internals)

### Memory / Performance

**How dict stores a value:**
```
hash("name")              → 8824780412431486381
8824780412431486381 % 8   → 3
store ("name", "alice") at slot 3

Internal array:
[None, None, None, ("name","alice"), None, None, None, None]
  0      1      2          3           4      5      6      7
```

Python stores the **key alongside the value** at every slot — this is how it handles hash collisions (explained below).

**Time Complexity:**
```
Insert → O(1)   hash + modulo = constant time
Lookup → O(1)   same calculation → direct slot access
Delete → O(1)   same calculation → go to slot → remove
```

**Hash Collision — when two keys land on the same slot:**
```python
hash("name") % 8 = 3
hash("mane") % 8 = 3   # different key, same slot!
```

Python uses **Open Addressing** to resolve this:
```
"name" → slot 3 empty → store here ✅
"mane" → slot 3 TAKEN → try slot 4 → empty → store here ✅

LOOKUP "mane":
→ slot 3 → stored key is "name" → ❌ not my key
→ slot 4 → stored key is "mane" → ✅ match → return value
```

**Rehashing — when dict gets full:**
```
Dict resizes at ~60% full (load factor)
8 slots → 5 keys → 60% full → resize to 16
        → recalculate ALL keys with % 16
        → move everything → this is called Rehashing

Why 60%? Beyond that, collisions become too frequent,
O(1) lookup starts degrading to O(n)
```

### CPython Implementation Detail

Python 3.7+ dict has two internal structures:
1. **Hash table** — for O(1) key lookup
2. **Linked list** — for tracking insertion order

Before 3.7, only the hash table existed → iteration order was random. After 3.7, insertion order is guaranteed — this fixed real production bugs in API responses where JSON field order was unpredictable.

## 🆚 Comparison / Trade-offs

| Scenario | Use This | Why |
|----------|----------|-----|
| Key-value data, dynamic keys | `dict` | Flexible, mutable |
| Need default value for missing keys | `defaultdict` | No KeyError, no if-checks |
| Counting frequency of items | `Counter` | Built-in counting + most_common() |
| Order-sensitive key equality check | `OrderedDict` | Equality considers order |
| Fixed structure, read-only record | `namedtuple` | Lighter than dict, immutable |
| Data + behavior together | `class` | Only option with methods |
| Size known upfront | dict comprehension | Pre-allocates, no rehashing |
| Size unknown, building incrementally | `{}` + loop | Fine, but ~20 rehashes for 1M keys |

## 💻 Code I Wrote Today

```python
# --- HASH TABLE INTERNALS ---

# hash() converts key to a number
print(hash("name"))   # 8824780412431486381
print(hash("age"))    # 2250463537539136114
print(hash("role"))   # 6818564456710138603

# modulo shrinks to valid slot index
print(hash("name") % 8)   # 3  ← slot 3

# Same key = same slot always
d = {"name": "alice"}
# insert: hash("name") % 8 = 3 → store at slot 3
# lookup: hash("name") % 8 = 3 → go to slot 3 → return "alice"


# --- SAME KEY OVERWRITES ---
d = {}
d["name"] = "alice"
d["name"] = "bob"
print(d["name"])   # "bob" — alice is overwritten


# --- WHY LIST FAILS AS KEY ---
d = {}
# d[[1, 2, 3]] = "value"   # TypeError: unhashable type: 'list'

# Proof of why:
key = [1, 2, 3]
# hash([1, 2, 3]) → FAILS because list is mutable
# If key changed after insert → Python can never find value again


# --- TUPLE AS KEY (WORKS) ---
d = {}
d[(1, 2, 3)] = "value"    # ✅ tuple is immutable → hashable
print(d[(1, 2, 3)])        # "value"

# BUT tuple containing list FAILS
# d[(1, [2, 3])] = "value"   # TypeError: unhashable type: 'list'
# hash((1, [2, 3])) → tries hash([2, 3]) → list is mutable → 💥


# --- REHASHING: COMPREHENSION VS LOOP ---
# ❌ Triggers ~20 resizes for 1M keys
d = {}
for i in range(1_000_000):
    d[i] = i * 2

# ✅ Pre-allocates correct size — zero resizing
d = {i: i * 2 for i in range(1_000_000)}


# --- DICT ORDERING (Python 3.7+) ---
d = {}
d["name"] = "alice"
d["age"] = 25
d["role"] = "engineer"

for key in d:
    print(key)
# name → age → role   (always insertion order) ✅


# --- DICT COMPREHENSION ---
numbers = [1, 2, 3, 4, 5]
d = {num: num * num for num in numbers}
print(d)   # {1: 1, 2: 4, 3: 9, 4: 16, 5: 25}


# --- defaultdict ---
from collections import defaultdict

# Without defaultdict — defensive code
d = {}
if "count" not in d:
    d["count"] = 0
d["count"] += 1

# With defaultdict — clean
d = defaultdict(int)
d["count"] += 1   # works directly ✅

# defaultdict with list — group users by city
users = [
    ("alice", "Mumbai"),
    ("bob", "Delhi"),
    ("charlie", "Mumbai"),
]

city_users = defaultdict(list)
for name, city in users:
    city_users[city].append(name)

print(dict(city_users))
# {"Mumbai": ["alice", "charlie"], "Delhi": ["bob"]}


# --- Counter ---
from collections import Counter

words = ["apple", "banana", "apple", "mango", "banana", "apple"]
counts = Counter(words)
print(counts)                  # Counter({"apple": 3, "banana": 2, "mango": 1})
print(counts.most_common(2))   # [("apple", 3), ("banana", 2)]

# On strings
text = "hello world"
count = Counter(text)
print(count.most_common(2))   # [('l', 3), ('o', 2)]


# --- OrderedDict ---
from collections import OrderedDict

d1 = {"a": 1, "b": 2}
d2 = {"b": 2, "a": 1}
print(d1 == d2)   # True — regular dict ignores order

od1 = OrderedDict([("a", 1), ("b", 2)])
od2 = OrderedDict([("b", 2), ("a", 1)])
print(od1 == od2)   # False — OrderedDict cares about order
```

## ⚠️ Gotchas I Discovered

1. **Tuple containing a list is NOT hashable** — immutability must go all the way down. `(1, [2, 3])` fails as a dict key even though tuple itself is immutable. Python checks every element recursively.

2. **Dict comprehension is faster than loop** — not just shorter code. When size is known upfront, comprehension pre-allocates the right array size and avoids ~20 rehash operations for large dicts.

3. **Dict ordering is Python 3.7+ only** — in Python 3.5/3.6 iteration order was random. Never rely on order if your codebase might run on older Python.

4. **defaultdict never raises KeyError** — even for keys that don't exist. Accessing a missing key auto-creates it with the default value. This can hide bugs if you're checking `if key in d` after the fact.

5. **Race condition with dicts** — dict is NOT thread-safe. In production with concurrent requests, two threads incrementing the same key can cause data loss. Needs locks or atomic operations (covered in Topic 17 — Concurrency).

## 🏭 Production Scenario

**Scenario 1 — API request tracking with Counter:**
```python
from collections import Counter

request_log = Counter()

# Every time a route is hit:
request_log["GET /users"] += 1
request_log["POST /login"] += 1
request_log["GET /users"] += 1

# Dashboard endpoint — top 5 busiest routes
top_routes = request_log.most_common(5)
```

**Scenario 2 — Grouping DB results with defaultdict:**
```python
# Raw DB results: [(user_id, order_id), ...]
from collections import defaultdict

user_orders = defaultdict(list)
for user_id, order_id in db_results:
    user_orders[user_id].append(order_id)

# Clean dict of user → [orders]
```

**Scenario 3 — Dict comprehension for building response:**
```python
# Transform list of user objects into id → name mapping
users = db.query(User).all()
user_map = {user.id: user.name for user in users}
# Fast O(1) lookup by ID anywhere in codebase
```

## 🎤 Interview Angle

**Q: How does Python dict lookup work? Why is it O(1)?**

**Surface answer** (junior):
Dict uses key-value pairs and Python finds the value using the key directly. It's fast because it doesn't loop through everything.

**Deep answer** (senior):
Python dict is built on a hash table. When you insert a key, Python runs `hash(key)` to get a large integer, then applies modulo with the current array size to get a valid slot index. The value is stored at that exact slot alongside the key.

On lookup, Python performs the same calculation — same key → same hash → same slot → direct access. No iteration. That's why it's O(1) regardless of dict size.

When two different keys hash to the same slot (collision), Python uses Open Addressing — it probes the next available slot and stores the key there. On lookup, it verifies the stored key matches before returning the value.

The dict resizes when ~60% full (load factor threshold) — it doubles the array and rehashes all existing keys. This is why dict comprehensions are preferred over loops when size is known upfront — they pre-allocate the right size and avoid multiple rehash cycles.
